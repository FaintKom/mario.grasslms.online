Phase 1 of 5

A · Analyse

Audience persona Performance context Constraints Prior-art scan

Audience. US high-school students preparing for the SAT Math section. Mixed prior-math confidence. Self-paced, mobile-first study habits.

Performance context. The digital SAT (April 2024 redesign onwards): roughly 38 questions in 70 minutes, adaptive section, calculator permitted throughout. Whole-task skill = decision under time pressure across multi-step problems.

Constraints. Solo build. No Articulate / Storyline seat. Widgets must render in any modern web LMS via iframe. Content must regenerate on demand when the SAT format shifts (because it does).

Prior-art scan. Khan Academy + College Board official prep + Magoosh — strong content libraries, weak on whole-task transfer practice under time pressure. Gap targeted.

Phase 2 of 5

D · Design

4C/ID Backward design Worked-example fading

Why 4C/ID over linear ADDIE.Design. SAT Math is a complex cognitive whole-task skill — multi-step problem solving under time. 4C/ID exists for exactly this transfer profile; ADDIE's structural strength (compliance sign-off per phase) was irrelevant for solo work.

Module sequence. Heart of Algebra as starter (lowest threshold to confidence; build a streak first) → Problem Solving + Data Analysis → Advanced Math → Additional Topics.

4C/ID component map.

  • Learning tasks — full SAT-shaped problem sets ordered easy → hard.
  • Supportive info — concept primers per skill block.
  • Procedural info — formula reference + Desmos technique cards delivered at the moment of need, not front-loaded.
  • Part-task practice — drill widgets per sub-skill (e.g. linear-equation manipulation) until automatic.

Authoring layer decision. JSON-first content separated from rendered HTML so any widget can regenerate when the format changes. This decision paid back in Develop.

Phase 3 of 5

D · Develop

Python generator Three-layer JSON Post-processing pipeline

The widget generator. sat_math_widgets.py — 63 KB Python — converts a task spec to an interactive Desmos / graph / formula widget HTML block. One generator handles all task variants.

Three-layer JSON content library.

  • sat_math_pro_course.json — 86 KB raw curriculum.
  • sat_math_pro_course_4cid.json — 185 KB 4C/ID-coded variant (which task is which component).
  • sat_math_pro_course_html.json — 675 KB fully rendered HTML widgets per task.

Normalisation pipeline. Three post-processing scripts before LMS upload: fix_interactive.py + fix_theory_final.py + fix_desmos.py. They catch JSON drift, repair broken Desmos expressions, and lock down rendering edge cases.

The big payoff. When the SAT switched to digital format (April 2024), I regenerated the affected widgets in days instead of rebuilding a Storyline course. The JSON-first decision in Design was the deciding bet.

Phase 4 of 5

I · Implement

HTML5 + Desmos JS LMS-agnostic Standalone preview

Output is the 675 KB HTML widget set, deployable as iframe content in any modern web LMS. Widgets are vanilla HTML5 + Desmos JS API — no SCORM dependency, though SCORM packaging can be layered on (same approach as the FinTech B2B SCORM module in this portfolio).

Standalone web preview also available so a reviewer can run a task without setting up an LMS — a deliberate pattern across this portfolio's artefacts.

Lessons — what I'd build again, what I'd change

  • JSON-first authoring paid back. When the SAT redesigned to digital format, regeneration took days vs a Storyline rebuild.
  • Heart-of-Algebra as starter reduced perceived difficulty in the first hour. Students built a streak before the harder modules.
  • Widget-generator pattern is transferable. The same generator framework powers Math Slides Add-on (Google Apps Script · JSX) and informs the LMS curriculum seeder used in GrassLMS.
  • Next iteration: emit xAPI statements per widget interaction to give L3 telemetry richer than mastery-point counts.
