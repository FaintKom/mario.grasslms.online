Phase 6 · Measure · feedback intake pipeline

Lesson Review Toolkit

I built this Python pipeline to turn scattered lesson-review feedback — Slack threads, Notion comments, mentor rubric forms, post-cohort survey CSVs — into a single normalised issue database that a design team can triage in standup. Same architecture I used on the AI Math EdTech video-lesson analysis pipeline.

Stack: Python 3.12 · Pandas · OpenAI structured outputs · JSON Schema Inputs: 4 source types · ~25 scripts · ~4 700 LOC Output: normalised JSON issue DB + HTML triage viewer
01

The pipeline · four stages

collect → extract → normalise → apply
Stage 01
Collect
Pull raw reviews from each source. Slack export, Notion API, mentor rubric Google Sheets, post-cohort survey CSV, Zoom call transcripts.
collect_reviews.py
Stage 02
Extract
Run each raw item through an LLM with a strict JSON-schema response. Extract issue text, severity, lesson reference, source ID, timestamp.
extract_content.py
Stage 03
Normalise
Cluster near-duplicates (cosine similarity on embeddings); collapse to a single canonical issue with a list of sources. De-dupe to one row per real problem.
normalize_items.py
Stage 04
Apply
Match each normalised issue against the lesson's content checklist. Emit a triage HTML page grouped by severity + lesson with one-click fix / defer / dismiss buttons.
apply_to_checklist.py

The pipeline is built so each stage emits a stable JSON intermediate. A new source can be added by writing only the Collect adapter — Extract, Normalise, and Apply do not change. A new LLM model is swapped in Extract without touching the other three stages.

02

Stage 02 · Extract · structured-output schema

what each raw review collapses into
# extract_content.py — schema passed to OpenAI structured outputs SCHEMA = { "type": "object", "properties": { "issue_text": {"type": "string", "description": "the problem in one sentence, reviewer's words preserved"}, "severity": {"type": "string", "enum": ["blocker", "major", "minor", "polish"]}, "category": {"type": "string", "enum": ["content", "pacing", "clarity", "interaction", "assessment", "tooling"]}, "lesson_id": {"type": ["string", "null"], "description": "e.g. M3.L2.S05; null if reviewer did not specify"}, "reviewer_role": {"type": "string", "enum": ["mentor", "peer", "learner", "sme", "unknown"]}, "suggested_action": {"type": ["string", "null"], "description": "reviewer's own fix if offered, else null"}, "source_meta": { "type": "object", "properties": { "source": {"type": "string", "enum": ["slack", "notion", "sheets", "survey", "transcript"]}, "source_id": {"type": "string", "description": "thread ID, page ID, row ID — for round-trip linking"}, "timestamp_iso": {"type": "string", "description": "ISO 8601 UTC"} }, "required": ["source", "source_id", "timestamp_iso"] } }, "required": ["issue_text", "severity", "category", "reviewer_role", "source_meta"] }

Schema-enforced LLM output means the Normalise stage can rely on the shape of every record. No defensive parsing, no try / except chains downstream.

03

Stage 03 · Normalise · canonical issue shape

post-dedup, sources collapsed
// example canonical issue after normalisation — what the triage UI consumes { "issue_id": "ISS-0142", "canonical": "Screen 4 talk-track is too dense — reviewers cannot find the opener inside the prop description.", "severity": "major", "category": "clarity", "lesson_id": "M1.L4.S04", "reviewer_mix": {"mentor": 2, "peer": 1, "learner": 4}, "first_seen": "2026-03-04T09:12:00Z", "last_seen": "2026-05-09T16:48:00Z", "sources": [ {"source": "slack", "source_id": "thread:C12345/p1234567", "text": "the props block runs into the opener, hard to scan mid-call"}, {"source": "sheets", "source_id": "mentor-rubric/row:48", "text": "Screen 4 opener buried"}, {"source": "survey", "source_id": "post-cohort-Q4/row:127", "text": "I cant find what to say on screen 4"} ], "suggested_action": "Move opener to its own line with bold trigger phrase; collapse prop description into a details disclosure.", "triage_state": "open" }

Embedding clustering collapses near-duplicate phrasing (e.g. "buried opener", "opener hidden", "cannot find what to say on screen 4") into a single canonical issue. The sources array preserves the original wording for audit — designers click through to the original Slack thread when triaging.

The Apply stage emits a single HTML page grouped first by severity (blocker → major → minor → polish), then by lesson. Each issue card shows the canonical text, source links, suggested action, and three buttons: fix (creates a Linear ticket), defer (moves to next-cohort backlog), dismiss (closes with a one-line reason that gets written back to the JSON).

Standup goes from "skim 472 Slack messages + 96 survey rows + 35 mentor rubric forms" to "triage 63 cards in 25 minutes." The pipeline is the difference between feedback that lives in chat scrollback and feedback that lands as a fix.

05

Where this sits in the program

Phase 6 · Measure · the intake half

This pipeline is the intake half of Phase 6 Measure. The output half is the Kirkpatrick L1–L4 plan — what to measure and how. Together they close the loop:

  • Kirkpatrick plan says what we want to know (e.g. "L2 quiz mastery ≥80 %", "L3 talk-track adherence ≥80 % in first 5 calls").
  • Lesson Review Toolkit is one of the instruments — qualitative feedback intake that complements the quantitative SCORM / xAPI telemetry.
  • Together they tell the team which iteration to ship next, not just whether the last cohort was happy.
