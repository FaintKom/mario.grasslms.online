Operationalises `D-002` — the curriculum spine is 4C/ID, not topic-by-topic. Inherits the six task classes from `01-analyze/task-inventory.md` §2 and the supportive / procedural / part-task component lists from §§3–6. # 4C/ID blueprint · whole-task curriculum architecture

## 1 · Whole-task definition

Verbatim from `01-analyze/task-inventory.md` §1:

> **Conduct a 60-second-opener-to-calendar-close cold call against an ICP buyer.**
>
> Operationally: from the moment the rep clicks "dial" in Outreach to the moment a calendar invite for a next-step meeting is sent and accepted, against a buyer matching one of the four ICP archetypes in brief §12.

The whole task is **configural** (per brief §9 A.S.: *"Top reps sound nothing alike. They don't follow a script. But those three moves are always there."*). 4C/ID is the only framework in `frameworks-applied.md` that prescribes a curriculum architecture for configural mastery — `D-002` is the binding decision.

---

## 2 · The four components · design summary

Van Merriënboer & Kirschner (2018) prescribe four components for any complex-skill program. This blueprint maps each onto the FinTechCard build:

| Component | Definition | Where it lives in this program |
|---|---|---|
| **(1) Learning tasks** | Whole-task experiences sequenced simple → complex across *task classes*. Variability inside each class. | The six 4C/ID task classes from `task-inventory.md` §2 → six modules. The solo-problem block of each module = the whole-task instance for that class. |
| **(2) Supportive information** | Mental models / strategies the learner activates to handle non-routine aspects of the task. | The 3-move model, ICP archetype model, 4-prop model, regulatory deflection table, brand-voice rules (§18.1). Delivered before / alongside learning tasks; referenced not memorised. |
| **(3) Procedural information** | Just-in-time how-tos for routine sub-skills (tool operations). | Delivered as mini-OS coach-marks (`D-007`) at point-of-use: Outreach dial → Salesforce log → Gong tag review → Calendar second-tab → LinkedIn scan. |
| **(4) Part-task practice** | Repetitive drill on automatable sub-skills until they free working memory for the configural task. | The six D1–D6 drills from `task-inventory.md` §6. Each module hosts its drill in the part-task block (≤ 5 min). |

## 3 · Task-class progression · 6 classes → 6 modules

Per Van Merriënboer & Kirschner (2018) Ch. 4, task classes ladder from simplest meaningful whole task to most complex, with explicit scaffolding fade. Variability inside each class is mandatory (Ch. 5).

### 3.1 · Class-by-class spec

| # | Class | Module | Cognitive demand | Scaffolding fade plan | Variability axis (which §12 archetypes) | Keystone moves practised (configural) |
|---|---|---|---|---|---|---|
| **1** | Warm inbound — buyer already interested | (scaffold inside M3) | Lowest. E1+E2+E5 (3 elements per `cognitive-load-analysis.md` §8). Rep practises M1+M2+M3 structure without rejection load. | Worked example (full) → completion (close-line redacted) → solo (mini-OS) | Maria (low-friction); used as a *Class 1 ramp* inside M3's worked-example slice — not its own module | M1 (light) + M3 (full) |
| **2** | Cold outbound to receptive buyer | M1 | 5 elements. Diagnostic creation is the new load. | Worked example → completion → solo | Maria (retail) + Tom (Series-B founder) — UK behavioural vs DE quant axis per §10.1 / §10.2 | M1 (full) + M3 (light) |
| **3** | Cold outbound to hostile buyer | M2 | 6 elements; objection inventory adds E6. | Worked example × 2 (UK Visa + DE Brex) → completion → solo | Maria ("Visa") + Tom ("Brex") + Emma ("accountant friction") | M1 + **M2 (full)** + M3 |
| **4** | Competitive displacement (Brex named) | (integrated in M2 + M4) | 7 elements; competitive frame E9 adds. | Worked example (L.D. GC-06) → completion → solo | Tom + (extended) Lukas trust variant | M1 + M2 (Brex variant) + M3 |
| **5** | Multi-stakeholder | M5 | 7 elements; second voice E10. | Worked example (Emma + accountant) → completion → solo (founder + finance) | Emma + Tom-dual-voice variant | M1 ×2 + M2 + M3 + Prop mapping |
| **6** | Regulatory-question deflection | M6 | 7 elements; reg overhead E7 + job-aid lookup. | Worked example × 5 acronyms (§19 row-per-row) → completion → solo (5 calls, 2 distractors) | Lukas (lead) + all 4 archetypes asking reg questions | M1 + M2 (reg-as-objection) + M3 + Reg deflection |

### 3.2 · Scaffolding fade (across modules)

4C/ID's fade is not within-module-only; it's *across* the program too. The rep gets more scaffolding in M1 (their first whole-task experience) and less in M6 (their last):

| Module | Worked-example detail | Completion redact size | Solo problem variability |
|---|---|---|---|
| M1 | Full transcript + annotation + Mayer signalling on every M-move | Single move redacted (M1 only) | One new archetype, same task class |
| M2 | Two worked examples (UK + DE) + annotation | M2-only redaction | Same archetype, hostile variant |
| M3 | Worked example side-by-side with mini-OS Calendar | Close-line redacted | Mini-OS habit-instrumented (Calendar must open pre-dial) |
| M4 | Four mini worked examples (one per archetype) | Archetype-match redacted | Three interleaved calls, full M1+M2+M3 |
| M5 | One worked example, multi-voice | Prop-pick redacted | Multi-stakeholder, both voices arrive |
| M6 | Five mini examples (one per acronym) | Deflection-row redacted | Five calls, 2 distractors + 1 escalate case |

Per Kalyuga's expertise reversal effect (Kalyuga 2007, `frameworks-applied.md` §8), top-decile reps (M.G., L.D. profile) need *less* worked-example. The Phase 6 iteration playbook surfaces adaptive sequencing (`06-iterate/iteration-playbook.md`); v1 ships a single track because cohort sizes (10–15) don't power per-rep adaptation.

### 3.3 · Variability of practice — module-by-module

Per Van Merriënboer (2018) Ch. 5, practice items within a class must vary on the dimensions that drive *schema generalisation*. Blocked practice (all Maria then all Tom) is rejected; interleaving is mandatory.

| Module | Variability dimension | Source of variance |
|---|---|---|
| M1 | UK behavioural-anchored vs DE quant-anchored diagnostic | §10.1 vs §10.2 — explicit cultural-frame split |
| M2 | 5 objection families | §10 + §12 spread: "we already have", fees, accountant friction, trust, Brex |
| M3 | Warm vs cold close stems | §11 close patterns: GC-01 (warm-ish) vs GC-15/16 (cold receptive) |
| M4 | 4 §12 archetypes, interleaved | Maria / Tom / Emma / Lukas — interleaved not blocked |
| M5 | Founder+finance vs owner+accountant pairing | §12.2 Tom + analyst vs §12.3 Emma + accountant |
| M6 | 5 §19 acronyms × 4 archetypes | KYC / AML / SCA / PSD2 / GDPR × Maria / Tom / Emma / Lukas, with explicit "do not deflect" distractors |

## 4 · Supportive information per class

Supportive info = the *frameworks* the rep references on non-routine aspects (Van Merriënboer & Kirschner 2018 Ch. 8). It is delivered before / alongside the learning task, *not* memorised — it lives in the rep's referenceable workspace.

| Mental model | Delivered in | Reference location post-launch | Phase 1 source |
|---|---|---|---|
| **3-move model (M1/M2/M3)** | M1 introduction; reinforced every module | Mini-OS pinned Slack channel + rep desktop 1-pager | brief §9 A.S.; §11 aggregate |
| **ICP archetype model (Maria/Tom/Emma/Lukas)** | Pre-training card (`curriculum-blueprint.md` §5) + M4 deep dive | Mini-OS sidebar widget; pre-dial preview | §12.1–§12.4 |
| **4-prop model (Spend control / Multi-user / Auto-FX / Receipt-sync)** | M5 introduction; referenced in M1 + M4 | Mini-OS sidebar widget | §13.1 |
| **Regulatory deflection table (5 rows × buyer Q × rep response)** | M6 entire module | Mini-OS pinned Slack channel + 1-page job aid on desktop | §19 |
| **Brand-voice rules (transparent · anti-jargon · specific · concedes weakness)** | Every worked-example transcript demonstrates the rules; named in M5 LO-PP.2 | Worked-example annotations | §18.1 |
| **Strategy: which prop against which objection** | M5 decision card | Job aid | M.G. §10.1 ("the framing of when to use which prop against which objection") |

**Mayer pre-training principle.** The 3-move model + ICP archetype card are delivered *before* M1 fires so the rep enters M1 with the supporting schemas already cued. Reduces extraneous load in the first whole-task encounter.

## 5 · Procedural information

Procedural info = the *how* of tooling, delivered just-in-time inside the workflow (Van Merriënboer & Kirschner 2018 Ch. 9). The §14 audit verdict — *"Tools learned in isolation, not in workflow"* — is the direct trigger for `D-007` (mini-OS) and this section.

All procedural info lives inside the mini-OS as coach-marks or context-sensitive sidebars. No standalone tooling module.

| Procedure | Trigger | Where delivered | Module that introduces it |
|---|---|---|---|
| Outreach dial sequence (queue · dial · log call disposition) | First time rep enters mini-OS Outreach app | Coach-mark overlay on the Outreach window | M1 pre-dial workflow |
| Salesforce next-step logging (`Next_Step_Booked__c = TRUE`) | After first successful calendar-close attempt | Coach-mark on Salesforce record window | M3 |
| Gong post-call self-tag review | After Module 1 solo-problem call completes | Coach-mark in mini-OS Gong app | M1 closing screen |
| LinkedIn / Companies House 90-second signal scan | Pre-call workflow, every cold module | Coach-mark + 1-page job aid | M1 + M2 + M4 |
| Calendar-in-second-tab habit | M3 part-task block + every cold module afterwards | Mini-OS Calendar app + event-log instrumentation | M3 |

## 6 · Part-task practice

Part-task drills (Van Merriënboer & Kirschner 2018 Ch. 10) automate sub-skills until they free working memory for the whole task. Each drill ≤ 5 minutes per module per `task-inventory.md` §6 (within the ≤ 10-min cohort budget — `cognitive-load-analysis.md` §7).

| Drill | Sub-skill automated | Cadence (when in week) | Timing | Module |
|---|---|---|---|---|
| **D1 · Diagnostic-question reframing** | LinkedIn snippet → ≤ 15-word diagnostic | Once during M1 part-task block (first half of W1) | 5 min · 10 cards | M1 |
| **D2 · Objection one-question reframe** | 10 objection phrasings → buyer-back restatements | M2 part-task block (W2) | 5 min · 10 cards | M2 |
| **D3 · Calendar-time stating** | Speak two slots in one sentence (no read-out) | M3 part-task block (W3) | 3 min · drill against transcript | M3 |
| **D4 · ICP archetype identification** | Signal-set → Maria/Tom/Emma/Lukas in < 5 s | M4 part-task block (W4) | 4 min · 20 cards | M4 |
| **D5 · Prop-to-objection mapping** | Objection card → prop + framing | M5 part-task block (W5) | 5 min · 15 cards | M5 |
| **D6 · Regulatory deflection one-liner** | KYC/AML/SCA/PSD2/GDPR question → §19 verbatim | M6 part-task block (W6) | 4 min · 5 cards | M6 |

**Drill scoring rule.** ≥ 95 % completion on first attempt qualifies as automated. Below 95 % triggers re-drill after the +7-day spaced-retrieval mini-quiz; the spaced quiz already targets the same outcomes so the re-drill amplifies retrieval-mode practice.

**Drill placement inside the 10-min module.** The drill runs *inside* the module's "Apply" slice — not as a separate session. Per `cognitive-load-analysis.md` §7 the slice plan is: ≈ 1 min activate + 3 min worked example + 2 min completion + 3 min solo. The drill substitutes for the completion-+-solo combination on drill-heavy modules (M4, M5, M6 where the card decks are larger).

## 7 · Configural mapping · task class → module → keystone moves

This is the explicit table the brief asks for: which Phase 1 task class drives which module, and which keystone moves the rep is practising (configural — multiple at once, not isolated).

| Task class | Module | M1 practised? | M2 practised? | M3 practised? | Additional moves |
|---|---|:-:|:-:|:-:|---|
| Class 1 — Warm inbound (scaffold) | inside M3 demonstrate | ✓ (light) | (n/a — no objection) | ✓ (full) | Calendar habit |
| Class 2 — Cold receptive | M1 | ✓ (full) | (n/a) | ✓ (light) | Signal extraction · diagnostic phrasing · 5-s silence |
| Class 3 — Cold hostile | M2 | ✓ | ✓ (full) | ✓ | Objection classify + restate + surface |
| Class 4 — Competitive displacement | distributed in M2 + M4 | ✓ | ✓ (Brex variant) | ✓ | Additive-not-replacement frame |
| Class 5 — Multi-stakeholder | M5 | ✓×2 (one per stakeholder) | ✓ | ✓ (multi-invitee) | Pain→prop mapping · dual diagnostic |
| Class 6 — Regulatory deflection | M6 | ✓ | ✓ (reg-as-obj) | ✓ | §19 verbatim deflection + escalate decision |

**Reading the table.** Every module practises *at least two* keystone moves simultaneously; Modules 4 + 5 + 6 practise all three plus the module's additional move. This is the operational definition of configural mastery (per `D-002`).

## 8 · How the modules also serve part-skill / whole-skill blend

Per Van Merriënboer (2018) Ch. 3, the 4C/ID instructor must decide for each task class: *whole-skill from the start* or *part-skill on the way to whole-skill*?

Decision per class:

| Class | Decision | Why |
|---|---|---|
| Class 1 | Whole-skill (scaffolded) | Low element count; whole is feasible |
| Class 2 | Whole-skill from the start | M1 module IS the whole-task experience; M3 is "light" but present |
| Class 3 | Whole-skill, M2 emphasised | M1 already automated from M1 module; the rep practises the whole call but M2 is the only un-automated move |
| Class 4 | Part-skill (objection variant within M2; competitive frame in M4) | The competitive frame is a *variant* of M2, not a new whole skill |
| Class 5 | Whole-skill with high element-interactivity prerequisite | Gated behind Classes 2 + 3 completion (`cognitive-load-analysis.md` §8) |
| Class 6 | Whole-skill, reg-deflection on top of automated M1+M2+M3 | The §19 deflection move slots into the existing whole-task structure |
