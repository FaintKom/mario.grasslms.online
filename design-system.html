§ 00

What's actually off.

3 problems · ranked

Not "two systems randomly mixed". The pattern is sharper: the main index.html is mid-migration — palette was switched to Lively, typography wasn't. Everything else either already follows Lively or is a legacy file that pre-dates the migration.

01
The big one · typography

Main page still loads Instrument Serif and uses it for every headline.

index.html imports Manrope + Geist Mono + Instrument Serif, then sets --serif: 'Instrument Serif' and applies it to .hero h1, .section-head h2, .modal-head h2, .foot h2, .eng-name, .edu-t, .flagship h2, .flag-desc, .mock-foot .stat .n, and many more. Italic serif <em> is the dominant accent gesture.

Block A–E and every pipeline artefact use Manrope 800 only, with a rotated sun-300 highlight as the accent gesture. Zero serif anywhere.

→ fix

Remove Instrument Serif from the font import. Replace every font-family: var(--serif) with font-family: var(--sans) + font-weight: 800. Replace every <em> italic-green pattern with the .hl rotated sun-300 highlight.

02
Component vocabulary mismatch

Buttons, shadows, and radii speak two different dialects.

Main page .btn: 1px outlined pill, border-radius: 999, mono uppercase 12px. Block-A .btn.primary: filled green block, border-radius: 14px, sans bold 13px, with the pop-shadow 0 4px 0 0 green-700 press effect.

Cards on the main page use border-radius: 4 with hard ruled borders. Lively cards use r-md (14px) to r-lg (18px) with soft shadow-sm. The two card languages don't blend — they look like two products.

→ fix

Use the canonical .btn / .btn.primary / .btn.secondary / .btn.ghost from block-A everywhere. Replace card borders with the Lively .block pattern: paper-2 bg, 1px ink-100 border, r-lg, optional shadow-sm.

03
Dark-mode leftovers in a paper system

Modal scrim, sticky-CTA, and modal-schematic still target dark backgrounds.

.scrim uses color-mix(in oklab, var(--bg) 85%, black) — on dark this read as a deeper black; on paper it goes muddy beige. .modal-schematic is hardcoded background: #0a0a08. .tweaks panel uses box-shadow: 0 20px 60px rgba(0,0,0,.4) — too heavy for light. The film-grain overlay was already correctly removed.

linkedin-banners.html is the only file untouched by the migration — full editorial dark, no Lively at all.

→ fix

Scrim → rgba(10,26,16,.55). Modal-schematic → drop the black, use paper with a dotted-grid pattern in ink-100. Tweaks panel → use shadow-md. Banners → full rebuild on paper.

§ 05

Patterns · the recipes.

how components combine

Three repeating recipes that hold the whole portfolio together. Use them, don't fork them.

① Page hero eyebrow → H1 → lede → meta

Every page opens the same way. Eyebrow positions the page in the system (BLOCK A · LESSON GENERATION). H1 carries the promise with one sun-300 highlight word. Lede expands the promise in 1–2 sentences with <b> tags doing emphasis via green-50 underline. Hero-meta closes with 3–4 fact cells under a top border. 

<p class="eyebrow"><b>BLOCK A</b> · LESSON GENERATION</p>
<h1>A teach-ready lesson, <em>assembled</em> from a brief.</h1>
<p class="lede">Pipelines I built take a fifty-line brief and ship a
<b>scaffolded lesson</b> through six validators in under five minutes.</p>
<div class="hero-meta"> … </div>

② The "moment of joy" one per page, never more

The rotated sun-300 highlight is the system's one piece of personality — it's what makes Lively lively. The rule: one per page, on the page-level H1. Section heads inside the page should NOT use it (otherwise it becomes noise). Body emphasis uses <b> with green-50 underline / green-50 bg padding, not the highlight.

③ The What/Why pairing always together

Every demo on every block page is explained in a sidebar with two blocks: What happens (green-100 label) and Why it matters (sun-100 label). Never use one without the other — the pairing is the whole point. "What" is the mechanic, "why" is the design argument. This is what makes the portfolio read as instructional design, not just dev demos.

§ 07

Rules · what to do, and not.

10 decisions, pre-made
Do

One highlight per page. The rotated sun-300 pill goes on the page H1, on one word. That's it. Section heads use plain Manrope 800.

Don't

Don't sprinkle highlights through the page. Two highlights = no highlight. It stops being a moment of joy and starts being decoration.

Do

Use <b> for body emphasis, styled as ink-900 + green-50 background or a 2px green-600 bottom border. Calm and consistent.

Don't

Don't reach for italic anywhere. No <em>-as-italic, no serif italic display, no italic system. The system is upright Manrope.

Do

Pick from the existing color scales. Greens, sun, coral, ink. Each scale has a clear semantic — green = primary/success, sun = highlight/warning, coral = error/gap, ink = text/borders.

Don't

Don't introduce new hues (purple, blue, teal). If you need a fifth semantic, name it first in this document and add a full 5-step scale. Drive-by colors create the drift you just fixed.

Do

Always pair what + why. Every demo gets a sidebar with green-100-labeled "What happens" and sun-100-labeled "Why it matters". That pairing is the portfolio's instructional-design tell.

Don't

Don't ship a demo without "why". A demo with only "what" reads as a dev portfolio. The pairing is what positions you as a methodologist.

Do

Filled buttons with pop-shadow. Primary CTAs are green-600 filled, r-md, with the 0 4px 0 0 green-700 press shadow. Tactile. Specific.

Don't

Don't use outlined pill buttons (the editorial pattern from the old main page). They read as a different product.

Do

Geist Mono is for labels and code only. Eyebrows, tags, meta rows, file paths, code blocks. Body and headings stay in Manrope.

Don't

Don't set body copy in mono. Tempting on a "developer-looking" demo page, but it kills readability past two paragraphs.

Do

Borders are 1px ink-100. Card borders, dividers, section rules. One thickness, one color, everywhere.

Don't

Don't use heavy borders or dark rules. The system reads as paper. Heavy rules push it back toward editorial / brutalist.

Do

Stay on the 4-based spacing scale. s-1 (4) → s-20 (80). Pad and gap come from these tokens, not arbitrary px values.

Don't

Don't write padding: 17px 23px. If you need a value between scale steps, the scale is right and the design is wrong.

Do

Code blocks use the 4-color palette. ink-300 for comments, green-300 for keywords, sun-300 for strings, coral-300 for numbers. On ink-900 bg, r-md.

Don't

Don't ship a code snippet with default browser styling. Code is a first-class component in this portfolio. Style it.

Do

Sun-300 highlight + brand-mark green dot are the system's signatures. Keep both on every page where you can — they carry the brand without a logo.

Don't

Don't add a wordmark or extra logo system. The brand-mark + name in the topbar is the entire identity layer.

§ 08

Long-form rules · markdown → HTML.

for the engagement build script

The engagement section (pipeline/engagement/*) is generated from Markdown by _build-md-to-html.py. Long documents need different typographic rules than UI screens — narrower max-width, looser line-height, semantic H1→H4 cascade. These are the rules the SHELL template must follow.

Container width & rhythm long-form readability

  • main.sheet — max-width: 820px (60ch reading optimum at 15px Manrope).
  • Body line-height: 1.65. Reading copy needs more air than UI copy (1.55).
  • font-size: 15px base. No smaller for body — these are reference docs.
  • Paragraph margin: 0.8em 0. Vertical rhythm tied to font-size, not fixed px.

Heading cascade H1 once · H2 sections · H3 inside · H4 = eyebrow

  • H1 — Manrope 800, clamp(34px, 5.2vw, 52px), letter-spacing -.02em. One per document. Followed by a 64×2px green-accent underline (the document's signature).
  • H1 <em> — rotated sun-300 highlight pill. One word per document, ideally the page's promise. Authored in markdown as *highlight*.
  • H2 — Manrope 800, 26px, top border 1px ink-100, margin: 56px 0 14px. Sections of the document.
  • H3 — Manrope 800, 17px. Subsections inside an H2.
  • H4 — Geist Mono UPPER, 11px, green-700 — functions as an eyebrow within content, not a heading per se. Use for labels like "context", "why it matters", "open questions".
  • H5 — Geist Mono UPPER, 11px, ink-faint — the smallest label level.

Blockquote · the "verbatim source" pattern used for client quotes

Long-form engagement docs cite stakeholder quotes verbatim. The blockquote is the visual signature for those moments — it's how a methodologist's voice survives the round-trip from interview to artefact.

  • Background green-50, left border 3px solid accent.
  • Italic body copy is acceptable here (blockquote is a conventional italic context). Do NOT use serif.
  • Attribution: <cite> in Geist Mono UPPER 11px ink-faint, with a leading em-dash.

Code blocks & inline code monospace data inside paragraphs

  • Inline <code> — Geist Mono 0.88em, ink-50 bg, green-700 color, 3px radius, 1px 6px padding.
  • Block <pre> — ink-50 bg, 3px accent left border, 12.5px Geist Mono, no syntax-highlighting in long-form docs (keeps the system simple; if needed, use the block-A code block dark style for product demos only).

Tables data rows in markdown

  • Thead — ink-50 bg, Geist Mono UPPER 10px, ink-dim, letter-spacing .08em.
  • Tbody td — 13.5px Manrope, line-height 1.5, 1px solid ink-100 row borders, vertical-align top.
  • No zebra striping. The clean row borders carry the rhythm.

Topbar & foot document navigation

  • Topbar — paper bg (NOT dark navy), sticky, 11px Geist Mono UPPER nav-links. Three back-arrows: ← Portfolio / ← Pipeline / ← Engagement. Each up one level. Breadcrumb shows current path.
  • Foot — Geist Mono UPPER 10px, ink-faint, 1px top border. Left: author + breadcrumb. Right: link back to site root.
  • Brand mark — In topbar use simplified M· in green-600 (not the full square logo). The square logo is reserved for the main portfolio + LinkedIn banners.

What NOT to do in the generator common drift sources

  • Don't load Instrument Serif. Manrope + Geist Mono only.
  • Don't use italic for display headings. H1–H3 are always Manrope 800 upright.
  • Don't use the dark navy topbar. The dark topbar is for printable artefacts (A4 sheets). Long-form docs use paper bg topbar.
  • Don't inline-style overrides in the body. If a one-off needs special treatment, add a class to the SHELL template; don't sprinkle inline styles in generated HTML.
  • Don't redefine :root per file. The SHELL template is the single source of truth for tokens used across engagement docs.

The Python script pipeline/engagement/_build-md-to-html.py encodes all of the above. To change the rules, edit the SHELL template constant in that script, then re-run python _build-md-to-html.py to regenerate every engagement HTML in one pass.
